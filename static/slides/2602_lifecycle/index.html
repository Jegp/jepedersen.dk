"If the organism carries a 'small-scale model' of external reality and of its own possible actions within its head, it is able to try out various alternatives, ... and in every way to react in a much fuller, safer, and more competent manner to the emergencies which face it."

Kenneth Craik, The Nature of Explanation, 1943

Kenneth Craik, British psychologist, 1943. Writing before modern computers existed. His claim: we reason by running small simulations in our heads. Science is the systematic improvement of those simulationsâ€”shared across people.

Science = improving shared inner models

  • A result is only science if it helps others predict the world better
  • In computational science, the model is the software
  • Software is executable knowledge
The goal is not the publication. It is whether someone else can absorb your model and make better predictions than before. When the model is a simulation or algorithm, the code IS the model. Not an appendix. Not supplementary material. The thing itself.

Knowledge regression

Bad software does not just fail to advance knowledge...

...it actively regresses it.

Others build on a broken model. Errors compound.

This is the central claim. If a bug in your code leads others to draw wrong conclusions, those wrong conclusions get cited, extended, and taught. The inner models of an entire field can move in the wrong direction. This is not hypotheticalâ€”it has happened in genomics, economics, psychology.

Code is only as useful as our ability to track, share, and preserve it. Version control is the foundation everything else builds on.

What is version control?

A complete, timestamped history of every change to your code

Like track-changes in a Word document â€” but for software

The standard tool is git

Most scientists have used track-changes or "final_v2_FINAL_revised.docx". Git is the principled version of that, designed for code. It records who changed what, when, and why.

Why does it matter for science?

Reproducibility

Every result is tied to an exact snapshot of the code that produced it

Collaboration

Multiple people can work on the same code without overwriting each other

Accountability

If a bug is found, you can trace exactly when it was introduced

The reproducibility point is the key one here. A published result should link to a specific git commit â€” a frozen snapshot of the code as it was at the time of submission. Without this, "see our GitHub" is not reproducibility.

Version control + dependencies = reproducibility

Neither alone is sufficient. You need both the code and the environment frozen. Together, they make a result that is genuinely reproducible in Craik's sense: someone else can run your model and absorb it into their own understanding. What is reproducibility?

Re-run

Can you get the same numbers?

Re-use

Can you adapt the model for your own question?

Re-build

Can you extend, improve, or discard it?

Most people think of reproducibility as re-running and getting the same number. That is necessary but not sufficient. Re-use and re-build is what Craik's framework demands: can the reader absorb your model into their own inner model and act on it?

The mathematics analogy

Would you accept a physics paper that said:

"Our equations predict X â€” but we won't show the math."

Then why accept computational science that hides its code?

This is the simplest way to put it to a non-CS audience. In physics or chemistry, equations are shown because they ARE the science. In computational science, code plays the same role.

Before we talk about the lifecycle, the audience needs to understand what libraries and dependencies areâ€”and why they are a scientific problem, not just a technical one.

What is a library?

Published, reusable pieces of software

Like citing a paper â€” except the citation runs

You write


import numpy as np
result = np.fft.fft(signal)

You rely on

~500,000 lines of tested, optimised code

NumPy, SciPy, pandas, scikit-learn â€” your analysis rests on these. They represent decades of engineering. The important point: you are responsible for what they do even if you did not write them.

Runtime dependencies

Your code needs specific versions of libraries to run correctly

Libraries change â€” sometimes silently

A result from 2020 may not reproduce in 2026

"It worked on my machine" is not just an inconvenience. It is a scientific reproducibility failure. A result that can only be produced on one machine at one moment in time is not a result anyone can build on.

What if Numpy changes to v2?

A change at any layer can change your results.

When you press "run", you are not just running your five lines of code. You are running a stack: your script on top of scipy on top of numpy on top of BLAS (a linear algebra library) on top of the OS. Each layer has a version. Each version can differ between machines or over time. A scipy update in 2024 changed the default behaviour of several statistical tests. If you re-ran a 2022 analysis without noticing, your p-values changed silently.

  • The entire stack â€” Python, libraries, system libs â€” described in one file
  • Checked into version control alongside your code
  • Anyone who runs nix develop gets a bit-for-bit identical environment
  • Your environment becomes reproducible for the same reason your code does
The key insight: with Nix, the environment is not a side-effect of installation instructions in a README. It is source code â€” versioned, reviewed, and reproducible. Just as we version-control the analysis script, we version-control the entire stack it runs on. Nix locks every dependency transitively, all the way down to system libraries. Docker pins the application layer but not the OS; conda pins Python packages but not system libs. Nix pins everything. I use this for my system at home. I have A CONFIGURATION FILE that IS my operating system.

Stage 2: First code

Scripts, notebooks â€” exploratory and messy

The inner model starts to be externalised

Version control from day one â€” even for messy code

The most common failure: people wait until the code is "clean" before version-controlling it. By then, the history of decisions is lost.

Stage 3: A library

Shared, documented, tested code

Others can build on your model

The model enters the scientific commons - but start privately

This is where knowledge is truly transferred. Not at the paper stage. A paper describes the library. The library IS the science.

Stage 4: Reproducible environment

Pin your dependencies

The model is frozen in time â€” and preserved

Results can be re-run in ten years

Docker, conda-lock, Nix flakes. At minimum: a locked requirements file. Do this before submission, not after.

Stage 5: The paper

The narrative around the model

Points to the runnable, versioned code

A paper without code is an incomplete model

The paper is the story. The code is the science. Both are necessary. Neither alone is sufficient.

Practical steps

  • Version control everything â€” git from day one, before it is clean
  • Share your code â€” open source, always; code without access is not science
    • Please don't over-engineer. Start simple.
  • Pin your dependencies â€” conda-lock, Nix, Docker before submission
  • Build on existing libraries â€” don't reinvent, improve what exists
  • Advocate â€” software contributions are scientific contributions
These are not heroic acts. They are the minimum for science that can be built on.
